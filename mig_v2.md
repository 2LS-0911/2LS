# Система сбора и хранения атомов 2LS — v2 (целевая архитектура)

> Замена mig.md. Цель: каждый случай, прошедший через приложение, гарантированно и без ручного труда превращается в знание, которое ускоряет и улучшает следующие ответы. Документ фиксирует целевую систему + план миграции с текущей.

---

## 0. Семь принципов v2 (что меняется и почему)

| # | Принцип | Что было в v1 | Что становится |
|---|---------|---------------|----------------|
| 1 | **Один источник истины** | Два мастера: `atoms_clean.jsonl` И MongoDB — уже разошлись (3 199 строк против 3 179 документов) | Мастер — только MongoDB `atoms`. JSONL — ежедневный экспорт-бэкап (артефакт), не источник |
| 2 | **Логируется каждая сессия, а не только решённая** | Диалог живёт в памяти браузера; в Mongo попадают только подтверждённые кейсы | Сессия пишется на сервере с первого сообщения. Брошенные сессии — главный сигнал слабости скилла, сейчас они теряются полностью |
| 3 | **Менеджер проверяет атом, а не кейс** | Менеджер аппрувит сырой кейс → потом администратор вручную конвертирует в атом (узкое горлышко, умрёт на масштабе) | Кейс автоматически конвертируется в черновик атома; менеджер видит уже готовый атом и жмёт одну кнопку. Цель: кейс → активный атом ≤ 48 часов |
| 4 | **Дубль — это подтверждение, а не мусор** | Дедупликации нет вообще | Перед вставкой — поиск похожего атома; найден → не дублируем, а инкрементим `occurrences` и поднимаем `confidence`. Повторённый кейс делает знание надёжнее |
| 5 | **Изменил контент — пересчитай вектор** | `enrich_atoms.py` обновляет content без переэмбеддинга → вектор ищет старый текст (тихий баг поиска) | Любое изменение поисковых полей (`content`, `symptoms`, `root_cause`, `title`) автоматически триггерит re-embed |
| 6 | **Качество проверяется на записи** | Фильтр `_is_useful_atom()` работает только на чтении — мусор лежит в базе и фильтруется при каждом запросе | Валидация на записи ($jsonSchema + python-валидатор): мусор не попадает в базу. Фильтр на чтении остаётся как страховка |
| 7 | **Эксплуатация оценивает атомы** | `ai_rating` кейса никуда не влияет | Рейтинг кейса распространяется на атомы, которые участвовали в ответе (через rag_trace). Атомы с плохой историей — на ревизию |

## 1. Коллекции MongoDB и их роли

| Коллекция | Роль | Ключевые поля |
|-----------|------|---------------|
| `atoms` | **Мастер базы знаний.** Только проверенное | схема v2 ниже |
| `atoms_draft` | Черновики атомов из кейсов, ждут ревью | то же + `draft_of_case` |
| `sessions` | Каждый диалог с первого сообщения | `session_id, service_id, vehicle, messages[], rag_trace[], status, tier` |
| `cases` | Закрытые кейсы (бывш. `cases_pending`) | как сейчас + `case_summary` (из маркера скилла), `atom_ids[]` |
| `brands_dict` | Справочник нормализации | `canonical, aliases[], models[], engine_codes[]` |
| `services`, `transactions` | Без изменений | — |

`search_log` отдельной коллекцией не нужен — RAG-трейс пишется внутрь сессии (`rag_trace[]`), аналитика пробелов — агрегацией по `sessions`.

## 2. Схема атома v2

```json
{
  "id": "haval_h6_4b15_p0171_0042",
  "schema_version": 2,
  "atom_type": "case | dtc | reference | procedure | oscillogram",
  "status": "active",                     // draft | pending_review | active | deprecated
  "title": "Haval H6 1.5T — P0171 + плавающий ХХ: подсос через клапан ВКГ",

  "vehicle": {
    "brand": "Haval",                     // строго canonical из brands_dict
    "model": "H6 Gen3",
    "engine_code": "GW4B15B",             // нормализован
    "year_range": [2021, 2025],
    "platform_twin": "Tank 300"           // опционально: где ещё применимо
  },

  "symptoms": ["плавающие обороты ХХ", "бедная смесь банк 1"],
  "dtc_codes": ["P0171"],
  "content": "Полное описание случая/знания…",
  "root_cause": "Разрыв мембраны клапана ВКГ → неучтённый подсос",
  "diagnostic_steps": ["1. Дым-тест впуска…", "2. …"],
  "false_hypotheses": ["MAF — исключён продувкой и подменой"],
  "solution": "Замена клапана ВКГ",
  "parts": [{ "name": "Клапан ВКГ", "part_number": "1014200XEC01" }],

  "ref_values": [                          // ТОЛЬКО структурно, не свободным текстом
    { "param": "LTFT на ХХ", "value": "+18 → +2", "units": "%", 
      "conditions": "прогретый, до/после замены", "measured_on": "live"  }
  ],                                       // measured_on: live | spec | type_typical

  "source": { "type": "service_case",      // service_case | video | pdf | manual_entry
              "case_ids": ["case_a1b2c3"], // все кейсы-подтверждения
              "service_ids": ["svc_soyuz"] },

  "verified": true,
  "llm_generated": false,                  // true для атомов, рождённых enrich-пайплайном
  "occurrences": 3,                        // сколько раз кейс повторился (дедуп-слияния)
  "confidence": 0.92,                      // f(occurrences, verified, рейтинги кейсов)
  "quality_score": 0.88,                   // скользящая оценка из эксплуатации (см. §7)

  "embedding": [ ... ],
  "embedding_model": "voyage-3",
  "embedded_at": "2026-06-15T…",           // контроль: embedded_at >= updated_at контента

  "created_at": "…", "updated_at": "…",
  "deprecated": { "reason": null, "replaced_by": null }
}
```

**Правила схемы (валидация на записи, $jsonSchema + python):**
- `brand` и `engine_code` обязаны существовать в `brands_dict` — иначе reject (нормализуй до записи).
- Для `atom_type: case` обязательны `root_cause` ИЛИ непустые `diagnostic_steps`.
- `content` ≥ 120 символов; маркеры мусора («лишь заголовком», «невозможно извлечь»…) — reject на записи.
- Каждый элемент `ref_values` обязан иметь `measured_on`. Значения с `measured_on: "type_typical"` бот выдаёт только с пометкой уровня (правило трёх уровней скилла).
- Атомы никогда не удаляются — только `status: deprecated` с причиной и ссылкой на замену.

## 3. Справочник нормализации `brands_dict`

Без него база фрагментируется («Хавал»/«Haval»/«HAVAL» — три разных бренда для фильтра RAG) и поиск по марке промахивается.

```json
{ "canonical": "Haval",
  "aliases": ["хавал", "хавейл", "haval", "хавэйл"],
  "models": [ { "canonical": "H6 Gen3", "aliases": ["х6", "h6 3", "аш6"] } ],
  "engine_codes": [ { "canonical": "GW4B15B", "aliases": ["4b15", "1.5t gw"] } ],
  "platform_links": ["Tank"] }
```

Применяется в трёх точках: (1) на входе сессии — нормализация полей vehicle; (2) при атомизации кейса; (3) при импорте из Потока А. Новые алиасы добавляются менеджером при ревью атомов («встретил новое написание — добавь алиас»).

## 4. Поток Б v2 — онлайн-кейсы от сервисов (главный поток)

```
Механик жмёт «Начать диагностику»
   ↓ POST /api/session/start  (списание кредита — как сейчас)
   ↓ создаётся документ в sessions (status: active)        ← НОВОЕ
Каждый ход диалога /api/chat:
   ↓ RAG-поиск (каскад §6) → ответ Gemini
   ↓ $push в sessions: { user_msg, bot_msg, rag_trace }    ← НОВОЕ
Механик завершает → экран confirm → POST /api/solve:
   ↓ парсер вынимает [CASE_SUMMARY] из последнего ответа    ← НОВОЕ (контракт со скиллом v2)
   ↓ документ в cases (status: pending) + sessions.status: solved
   ↓ АВТО-АТОМИЗАЦИЯ (фоновая задача, сразу):               ← НОВОЕ
        case → LLM (Gemini Flash) по жёсткой схеме v2 → atoms_draft
        → нормализация по brands_dict
        → ДЕДУП-ПРОВЕРКА (§5)
        → эмбеддинг черновика
Менеджер: GET /api/manager/drafts → видит ГОТОВЫЙ АТОМ рядом с исходным кейсом
   ↓ [Одобрить] → atoms (status: active)  — одна кнопка
   ↓ [Править]  → инлайн-правка → re-embed → active
   ↓ [Отклонить]→ draft в архив с причиной (учим авто-атомизацию на отказах)
```

**Что даёт логирование сессий с первого сообщения (п.2 принципов):**
- Брошенные сессии (нет активности 24 ч → cron ставит `status: abandoned`) = карта провалов скилла. Сейчас эти данные теряются полностью — а это самый честный сигнал качества.
- `rag_trace` каждого хода = какие атомы реально находились и с какими score → аналитика пробелов (§8) и оценка атомов из эксплуатации (§7).
- Полная история нужна для разбора жалоб («бот посоветовал не то») — без серверного лога разбор невозможен.

**Контракт [CASE_SUMMARY]:** скилл v2 в конце подтверждённого диалога выдаёт служебный блок `[CASE_SUMMARY]{json}[/CASE_SUMMARY]` (vehicle, dtc, root_cause, parts, ref_values_measured, false_hypotheses, outcome). Парсер `/api/solve` кладёт его в `cases.case_summary` — авто-атомизация стартует с уже структурированных данных, а не с сырой переписки. Если маркера нет (старые диалоги, сбой) — атомизация работает по полной истории сообщений, просто чуть хуже.

**Кейсы с `no_answer: true` — не выбрасывать:** из них рождается атом-«антикейс» (`false_hypotheses` = что исключили) ИЛИ запись в отчёт пробелов. Что проверено и НЕ помогло — тоже знание.

## 5. Дедупликация-как-подтверждение (перед каждой вставкой атома)

```
Кандидат-атом готов (черновик с эмбеддингом)
   ↓ $vectorSearch по atoms: filter { vehicle.brand, atom_type }, top-5
   ↓ есть атом с cosine ≥ 0.93?
        НЕТ → вставляем как новый (pending_review)
        ДА  → сравнить root_cause (LLM-проверка «то же ли это знание», ответ да/нет)
              СОВПАДАЕТ → НЕ вставлять дубль:
                 $inc: { occurrences: +1 }
                 $push: { source.case_ids: новый кейс }
                 confidence = min(0.99, base + 0.05 × occurrences)
                 при расхождении деталей (другой артикул, другой год) — merge полей
              НЕ СОВПАДАЕТ (похожий симптом, другая причина) →
                 вставить новый + связать оба: related_atoms[] 
                 (две причины одного симптома — это ветвление, ценность для диагностики)
```

Эффект: база не распухает от повторов, а каждое повторение делает ответ увереннее. Бот сможет говорить: «эта причина подтверждена в N случаях по этой модели».

## 6. Поисковый каскад при /api/chat (использует новые поля)

```
Ступень 0. Нормализация запроса: brand/model/engine через brands_dict,
           извлечение DTC регексом [PBCU][0-9]{4}
Ступень 1. Точный матч: dtc_codes + brand (+engine_code если есть) → топ-5
Ступень 2. $vectorSearch (150 кандидатов) с pre-filter по brand
           → топ-20 → write-time мусора уже нет, но _is_useful_atom как страховка
Ступень 3. Если по точной модели пусто → расширение по platform_twin
           (Tank → Haval, Москвич → JAC…) с пометкой в контексте:
           «данные платформенного близнеца»
Ранжирование финальных топ-3:
   score = vector_score × W(atom_type)        // case выше reference
         × (0.7 + 0.3 × confidence)           // подтверждённые выше
         × quality_penalty(quality_score)      // плохая история — вниз
Пусто на всех ступенях → в контекст Gemini флаг no_kb_data
   (скилл включает честную формулу базы + активацию знаний по марке)
```

## 7. Петля качества: эксплуатация оценивает атомы

После закрытия кейса с `ai_rating`:
```
для каждого atom_id из rag_trace ходов этой сессии:
    $push: { usage_history: { case_id, rating, date } }
    quality_score = скользящее среднее последних 20 использований (нормировано 0–1)
```
- `quality_score < 0.5` при ≥ 5 использованиях → атом во вкладку «На ревизию» менеджеру.
- Еженедельный отчёт: топ-10 лучших и худших атомов недели.
- Атом, который регулярно participates в 5-звёздочных кейсах, получает буст в ранжировании (§6) — база самонастраивается на то, что реально помогает.

## 8. Аналитика пробелов (агрегация по sessions, еженедельный cron)

Отчёт «Пробелы недели» менеджеру:
1. **Запросы без покрытия:** ходы, где max(rag_trace.score) < порога → группировка по brand+model+система → топ-20.
2. **Брошенные сессии:** status: abandoned — на каком ходу бросили, какая марка, что спрашивали.
3. **no_answer-кейсы:** где скилл честно сдался.
4. **Низкие рейтинги:** кейсы с ai_rating ≤ 2 + их rag_trace.

Этот отчёт = план пополнения следующей недели (вместо догадок). Когда дойдёт до покупных баз (план v3) — приоритет закупки тоже строится из этого отчёта.

## 9. Поток А v2 — офлайн-источники (PDF/видео/осциллограммы)

Пайплайн остаётся (`pipeline.py` → enrich → import), три исправления:

1. **Мастер — Mongo:** `pipeline.py` пишет черновики сразу в `atoms_draft` (через тот же валидатор и дедуп §5). `atoms_clean.jsonl` больше не редактируется руками и не является источником — он генерируется командой экспорта (см. §10).
2. **enrich → re-embed обязателен:** `enrich_atoms.py` после `$set` контента ставит `embedded_at: null`; `import_mongo.py` пересчитывает вектор всем, у кого `embedded_at < updated_at` (а не только тем, у кого embedding отсутствует). Это закрывает текущий тихий баг.
3. **Write-time валидация:** «пустые» reference-атомы (content < 120, нет root_cause) не попадают в `atoms` вообще — живут в `atoms_draft` до обогащения. В поиске участвует только `atoms`.

## 10. Бэкапы и экспорт

- Ежесуточный cron: `mongoexport` коллекций `atoms`, `cases`, `sessions` → `backups/YYYY-MM-DD/` (+ ротация 30 дней).
- `atoms_clean.jsonl` — генерируемый артефакт: `python export_atoms.py` → выгрузка `atoms` (без embedding) в JSONL. Нужен для аудита глазами и переноса — но никогда не редактируется руками.
- Перед любой миграцией/массовой операцией — снапшот обязателен.

## 11. План миграции с v1 → v2 (порядок внедрения)

| Шаг | Что делаем | Зачем сначала | Оценка |
|-----|-----------|---------------|--------|
| 1 | Создать `brands_dict`, прогнать нормализацию по всем 3 179 атомам | Всё остальное (дедуп, фильтры, атомизация) зависит от чистых brand/engine | 0.5–1 день |
| 2 | Миграционный скрипт атомов на схему v2 (добавить status/confidence/occurrences/source; verified=true старым проверенным; ref_values из строки → структура где можно) | Поля нужны каскаду §6 и петле §7 | 1 день |
| 3 | Включить запись `sessions` в /api/chat (+ rag_trace) и cron abandoned | Каждый день без этого = потерянные данные. Самый срочный шаг | 0.5–1 день |
| 4 | Парсер [CASE_SUMMARY] в /api/solve (скилл v2 уже выдаёт маркер) | Дешёвый шаг, сразу улучшает кейсы | 0.5 дня |
| 5 | Авто-атомизация: case → draft → дедуп → embed; экран менеджера «черновики атомов» с кнопкой Одобрить | Снимает ручное горлышко до прихода потока кейсов | 2–3 дня |
| 6 | Re-embed-фикс в enrich/import + write-time валидация + переезд мастера в Mongo | Закрывает баг поиска и двоевластие источников | 1 день |
| 7 | Петля качества (§7) + cron «Пробелы недели» (§8) | Полезно с первых недель эксплуатации, но не блокирует запуск | 1–2 дня |

**Минимум для запуска в сервисах (нельзя выпускать без):** шаги 1–4. 
**Желательно до конца первого месяца:** шаги 5–6. Шаг 7 — по мере потока.

## 12. Метрики здоровья системы (панель менеджера)

- Кейсов за неделю / конверсия кейс → активный атом (цель ≥ 70%)
- Медианное время кейс → активный атом (цель ≤ 48 ч)
- Доля ходов с покрытием из базы (max score ≥ порога) — растущий график = база работает
- Доля abandoned-сессий (цель — снижение от недели к неделе)
- Средний ai_rating / доля no_answer
- Атомов active / pending_review / на ревизии по quality_score

## Сквозная схема v2 (end-to-end)

```
ПОТОК А (офлайн)                    ПОТОК Б (онлайн, главный)
PDF/видео/осц.                      Механик → session/start → sessions(active)
  ↓ pipeline.py                        ↓ /api/chat × N ходов
atoms_draft ←──── валидатор ────→      ↓ каждый ход: rag_trace в сессию
  ↓ enrich (re-embed!)                 ↓ /api/solve + [CASE_SUMMARY]
  ↓                                  cases (pending) → АВТО-АТОМИЗАЦИЯ
  └────────→ ДЕДУП §5 ←──────────────── atoms_draft (готовый черновик)
                ↓                            ↓
        atoms (active) ←── менеджер: одна кнопка «Одобрить»
                ↓
        $vectorSearch каскад §6 → следующий механик получает ответ
        быстрее и точнее ← confidence/occurrences/quality_score
                ↓
        ai_rating кейса → петля качества §7 → база самонастраивается
```

Главный сдвиг против v1: **система замкнута**. Каждый случай либо усиливает существующий атом (дедуп-подтверждение), либо рождает новый (авто-атомизация), либо помечает пробел (аналитика). Ручной труд остаётся в одной точке — кнопка «Одобрить» у менеджера на готовом атоме.
